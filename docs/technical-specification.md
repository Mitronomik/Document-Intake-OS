# Техническое задание: Локальная программа распознавания документов, ведения базы водителей и формирования заявок для трех терминалов

**Версия 1.0**  
**Дата:** 15 июля 2026 года

## 1. Назначение документа
Настоящее техническое задание описывает реализацию полностью локальной настольной программы для приема фотографий документов, распознавания и проверки реквизитов, ведения базы водителей и транспорта, подготовки JPEG-файлов до 2 МБ и формирования Excel-заявок по трем реальным шаблонам терминалов.

Основание: предоставленные фотографии реальных документов и файлы `TSPMAINFILE.xls`, `visitors_example.xlsx`, `MGSMAINFILE.xlsx`. Персональные данные из примеров в ТЗ не воспроизводятся.

## 2. Цели и границы проекта
- Сократить ручной ввод и повторное копирование паспортных и транспортных данных.
- Привести исходные фотографии к формату сайта: JPEG, один файл на логический документ, размер не более 2 МБ.
- Сохранить подтвержденные данные в локальной базе для повторного использования.
- Формировать точные копии трех Excel-шаблонов с данными выбранных водителей и транспорта.
- Оставить подачу заявки в «Конверсту»/на сайт ручной; прямая API-интеграция и браузерная автоматизация в MVP не входят.

## 3. Термины
| Термин | Определение |
|---|---|
| Партия загрузки | Набор исходных файлов, полученных и загруженных оператором за один рабочий сеанс. |
| Логический документ | Один документ водителя или транспорта, который может состоять из одного или нескольких исходных фото. |
| Оригинал | Неизмененная копия файла, загруженного оператором. |
| Подготовленный файл | JPEG, созданный из оригинала для загрузки на сайт: обрезанный, выровненный, при необходимости склеенный и размером не более установленного лимита. |
| Черновик OCR | Автоматически распознанные значения до подтверждения оператором. |
| Подтвержденные данные | Значения, проверенные оператором и сохраненные в рабочую базу. |
| Адаптер терминала | Набор правил заполнения конкретного Excel-шаблона, проверки комплектности и формирования выходной папки. |
| Снимок заявки | Неизменяемая копия данных, использованных при конкретном экспорте. |

## 4. Исходный и целевой процессы
### 4.1. Исходный процесс
1. Водитель присылает отдельные фотографии документов, часто без обрезки и выравнивания.
2. Оператор вручную определяет типы документов, разделяет или склеивает фото, повышает контраст и сжимает файлы.
3. Оператор перепечатывает данные в одну из трех таблиц.
4. Excel и JPEG загружаются на сайт вручную.
### 4.2. Целевой процесс
1. Оператор загружает новую партию фотографий.
2. Программа выделяет документы, предлагает тип, страну, сторону и владельца.
3. OCR, MRZ и штрихкоды формируют черновик полей.
4. Оператор проверяет сомнительные и критические значения.
5. Подтвержденные данные сохраняются в локальную базу.
6. Оператор выбирает терминал, водителей, тягач и прицеп.
7. Программа проверяет комплектность, формирует JPEG, Excel и папки водителей.

## 5. Функциональные требования
| ID | Функция | Требование |
|---|---|---|
| FR-01 | Создание партии | Оператор создает новую партию и загружает один или несколько файлов. Система присваивает партии уникальный номер, дату, автора и статус. |
| FR-02 | Прием файлов | MVP принимает JPG/JPEG, PNG и HEIC/HEIF. Допустимо добавить WEBP и PDF в следующем релизе. Исходники копируются в защищенное хранилище без изменения. |
| FR-03 | Контроль дублей | Для каждого файла вычисляются SHA-256 и перцептивный хеш. Система предупреждает о повторной загрузке того же или визуально похожего изображения. |
| FR-04 | Анализ качества | Проверяются ориентация, резкость, блики, контраст, обрезанные края, минимальное разрешение и наличие документа в кадре. |
| FR-05 | Выделение документов | Система обнаруживает один или несколько документов на фото. Оператор может изменить рамки, разделить кадр вручную или оставить как единый документ. |
| FR-06 | Классификация | Для каждой области определяется страна, тип документа, сторона и версия шаблона. Неуверенная классификация выводится на подтверждение. |
| FR-07 | Предобработка | Выполняются поворот, перспективное выравнивание, обрезка, мягкая коррекция контраста/резкости и удаление лишнего фона без уничтожения защитных элементов. |
| FR-08 | OCR и машинные зоны | Печатный текст распознается локально; MRZ и штрихкоды обрабатываются отдельными модулями; контрольные цифры используются для валидации. |
| FR-09 | Рукописные документы | Для миграционных карт система распознает печатные реквизиты, пытается прочитать рукопись и подставляет совпадающие данные из паспорта. Поля не считаются подтвержденными без оператора. |
| FR-10 | Связывание документов | Документы объединяются в карточку водителя и карточки тягача/прицепа. Система предлагает существующего водителя по ФИО, дате рождения и номеру документа. |
| FR-11 | Экран проверки | Оператор видит изображение и поля рядом. Нажатие на поле увеличивает соответствующую область. Значения маркируются зеленым, желтым или красным по уверенности и результатам проверок. |
| FR-12 | Подтверждение | Запись не становится рабочей до подтверждения оператором. Каждое исправление сохраняет автора, время, исходное и новое значение. |
| FR-13 | База данных | Система хранит водителей, документы, транспорт, прицепы, партии, заявки, версии шаблонов, файлы и историю изменений. |
| FR-14 | Выбор терминала | При создании заявки оператор сначала выбирает один из трех терминалов. После этого применяются его шаблон, допустимые значения и правила комплектности. |
| FR-15 | Выбор участников | Доступны режимы «только текущая партия» и «вся база». Выбираются водители и назначенные им на эту заявку тягач и прицеп. |
| FR-16 | Проверка комплектности | До экспорта проверяются обязательные документы, сроки действия, наличие сторон, совпадение ФИО/номеров, лимит участников и терминальные ограничения. |
| FR-17 | Формирование JPEG | На каждый логический документ создается один подготовленный JPEG. Лицевая и оборотная стороны склеиваются в заданном порядке; несколько документов из одного кадра разделяются. |
| FR-18 | Ограничение размера | Каждый подготовленный файл должен быть не более 1,90 МиБ, чтобы иметь запас относительно лимита сайта 2 МБ. Сжатие выполняется итеративно из оригинала. |
| FR-19 | Формирование Excel | Система создает копию реального шаблона выбранного терминала, очищает только область данных и заполняет строки в точном порядке столбцов. |
| FR-20 | Выходной комплект | Создается папка заявки с Excel-файлом, подпапками водителей, JPEG-документами, машинным реестром файлов и отчетом предупреждений. |
| FR-21 | История и повторный экспорт | Любую заявку можно открыть, посмотреть снимок данных и сформировать повторно без изменения первоначальной версии. |
| FR-22 | Резервное копирование | Администратор создает зашифрованную резервную копию базы и файлов и может восстановить ее на том же или новом рабочем месте. |
| FR-23 | Настройки шаблонов | Версии Excel-шаблонов и правила терминалов хранятся отдельно от кода и могут обновляться через административный экран. |

## 7. Каноническая модель данных
### 7.1. Сущности
- `UploadBatch`: партия загрузки, пользователь, время, статус.
- `SourceFile`: оригинальный файл, хеши, метаданные, оценка качества.
- `Document`: тип, страна, версия, стороны, подготовленный JPEG, статусы OCR и проверки.
- `Person`: кириллические и латинские варианты ФИО, дата/место рождения, пол, гражданство, телефон.
- `IdentityDocument`: тип, серия, номер, полное значение, даты, орган выдачи, код подразделения, MRZ, персональный номер.
- `MigrationDocument`: серия/номер, даты пребывания, штампы и связь с паспортом.
- `Vehicle`: роль тягач/прицеп, госномер, VIN, марка, модель, год, цвет, тип, массы, собственник.
- `Application`: терминал, выбранные участники и транспорт, версия шаблона, снимок данных, статус экспорта.
- `AuditEvent`: кто, когда и что изменил.

### 7.2. Правила хранения значений
- Хранить отдельно исходное OCR-значение, нормализованное значение и подтвержденное значение.
- Внутренние даты хранить в ISO `YYYY-MM-DD`, в Excel писать как настоящие даты.
- Номера документов, телефоны, VIN и госномера хранить как строки.
- Отдельно хранить ФИО кириллицей и латиницей; MRZ не использовать как источник кириллического написания.
- Не достраивать отчество или адрес, если данных нет.

### 8.4. Специальные правила visitors_example.xlsx
- В одной заявке допускаются посетители только с одинаковым гражданством. При смешанном выборе программа автоматически предлагает разбить заявку.
- Для иностранных граждан в колонке G выбирается «Национальный паспорт».
- Организация заполняется только если отличается от организации-заявителя; ИНН допустим только при заполненной организации.
- Для пешеходного пропуска транспортные поля R–X должны быть пустыми.
- При нескольких ТС у одного посетителя создаются отдельные строки с повторением ФИО и даты рождения.
- Примерная строка с Пушкиным удаляется; лист `Types`, комментарии и проверки данных сохраняются.
### 8.5. Специальные правила MGSMAINFILE.xlsx
- Рабочий лист экспорта — `Данные`; активные поля A–Y, резервные Z–AD пустые.
- Книга содержит внешнее Power Query-подключение. В сформированной копии обновление должно быть отключено или соединение удалено, чтобы данные не перезаписывались.
- Таблица должна расширяться до количества экспортируемых строк и сохранять оформление.
### 8.6. Специальные правила TSPMAINFILE.xls
- Рабочий лист — `ТСП`; данные начинаются со строки 2.
- Серия и номер паспорта записываются отдельно и как текст.
- Формат `.xls` сохраняется, если он обязателен для терминала. На Windows это рекомендуется делать через установленный Microsoft Excel.
- Лист `Лист1` не используется как источник текущей заявки и не изменяется, если отдельное терминальное правило не требует иного.

## 9. Обработка изображений
- Оригинал никогда не перезаписывается.
- Система учитывает EXIF-ориентацию, исправляет перспективу и дает оператору ручные рамки.
- При двух документах в кадре создаются два логических документа.
- При двух сторонах одного документа создается один JPEG: лицевая сторона первой, оборотная второй; направление склейки настраивается.
- Итог: JPEG, RGB, sRGB, без EXIF, без альфа-канала, не более 1,90 МиБ.
- Сначала меняется качество JPEG, затем при необходимости разрешение. Все попытки выполняются из оригинала, а не из ранее сжатой копии.
- Если лимит достижим только при потере читаемости, экспорт блокируется и предлагается запросить новое фото.

## 10. Интерфейс
1. **Главная:** новые партии, незавершенные проверки, последние заявки.
2. **Загрузка:** перетаскивание файлов, статус качества и дублей.
3. **Разделение кадра:** рамки документов, типы, стороны, поворот.
4. **Проверка OCR:** изображение слева, поля справа, масштабирование области.
5. **Карточка водителя:** документы, сроки, история, назначенный транспорт.
6. **Транспорт:** тягачи, прицепы, собственники и документы.
7. **Конструктор заявки:** терминал, фильтр текущей партии, выбор водителей и транспорта.
8. **Проверка комплектности:** ошибки, предупреждения, документы и сроки.
9. **Экспорт:** место сохранения, предпросмотр имен файлов и итоговый отчет.
10. **Администрирование:** шаблоны, справочники, резервные копии, пользователи.

## 12. Архитектура и технологии
- Настольное приложение: Python 3.12 + PySide6.
- Изображения: OpenCV + Pillow.
- OCR: PaddleOCR или эквивалентная локальная модель; инференс через локальный runtime.
- MRZ: отдельный детектор и парсер ICAO с проверкой контрольных цифр.
- Штрихкоды: ZXing-C++ или эквивалент.
- База: SQLite с SQLCipher или иной проверенной схемой шифрования.
- Excel `.xlsx`: библиотека, сохраняющая структуру шаблона; `.xls`: Microsoft Excel COM на Windows либо согласованный конвертированный шаблон.
- Код разделяется на domain, OCR adapters, image pipeline, terminal adapters, persistence, UI и tests.

## 13. Безопасность
- Все реальные документы обрабатываются только локально.
- База и файловое хранилище шифруются; ключ не хранится рядом с базой в открытом виде.
- Роли: оператор и администратор; автоматическая блокировка при бездействии.
- Журнал содержит идентификаторы сущностей и действия, но не полные паспортные номера.
- Резервные копии шифруются.
- Реальные документы запрещено добавлять в GitHub, Codex Web, CI и демонстрационные тесты.

## 15. Ошибки и статусы
- `NEW` — файл загружен.
- `NEEDS_SEGMENTATION` — требуется разделение/обрезка.
- `NEEDS_CLASSIFICATION` — неизвестен тип документа.
- `OCR_READY` — черновик распознан.
- `NEEDS_REVIEW` — есть неподтвержденные или конфликтующие поля.
- `VERIFIED` — оператор подтвердил документ.
- `INCOMPLETE` — не хватает документа или стороны.
- `READY_FOR_EXPORT` — комплект соответствует терминалу.
- `EXPORTED` — создан снимок заявки и выходной пакет.

## 17. Тестирование
- Unit-тесты нормализации дат, номеров, гражданств и терминальных преобразований.
- Golden-file тесты: ожидаемый Excel сравнивается по листам, заголовкам, значениям, форматам и структурным элементам.
- Набор обезличенных изображений: разные страны, поколения документов, фоны, углы, блики, размытие, один/два документа в кадре.
- Отдельная локальная приемка на реальных данных вне Git.
- Минимум 30 примеров каждого массового класса/версии документа до заявления о стабильной точности.
- Обязательная проверка загрузки всех трех выходных книг на реальных терминалах.

## 18. Этапы реализации
1. **Аналитика и фиксация правил, 1–2 недели:** терминалы, поля, документы, макеты экранов, приемочные примеры.
2. **Ядро без OCR, 2–3 недели:** база, партии, ручной ввод, подготовка JPEG, папки, три Excel-адаптера.
3. **OCR паспортов и транспорта, 3–5 недель:** MRZ, классификация, зональный OCR, экран проверки.
4. **Миграционные карты и пилот, 2–4 недели:** подстановка из паспорта, реальные ошибки, улучшение шаблонов.
5. **Промышленная доводка, 3–6 недель:** безопасность, резервное копирование, установщик, регрессия шаблонов.
Ориентир рабочего Windows-MVP: 8–12 недель; стабильная версия после пилота: 3–5 месяцев.

## 19. Вне объема MVP
- Автоматическая отправка заявок в «Конверсту» или обход ее интерфейса.
- Проверка подлинности документа как криминалистическая экспертиза.
- Гарантированное чтение любого рукописного текста.
- Общая облачная база и удаленный доступ через интернет.
- Большая локальная LLM/VLM без отдельного обоснования.

## 20. Вопросы, требующие подтверждения
- Уточнить официальное название третьего терминала, использующего visitors_example.xlsx.
- Подтвердить, обязан ли TSP принимать именно .xls или допускается предварительно согласованный .xlsx.
- Подтвердить максимальное количество водителей в одной заявке для каждого терминала; значение «не более 10» из рабочей схемы должно быть оформлено как терминальное правило.
- Подтвердить, должны ли изображения физически вставляться в Excel или достаточно папки документов рядом с Excel. В настоящем ТЗ базовым считается второй вариант.
- Передать матрицу обязательных документов по терминалу, гражданству, виду пропуска и типу транспорта.
- Подтвердить требуемую схему склейки: вертикально или горизонтально, порядок сторон и допустимые поля вокруг документа.
- Определить, нужна ли первая версия для macOS или достаточно Windows 11.
- Определить количество рабочих мест: один компьютер или несколько операторов в локальной сети.
- Определить срок хранения оригиналов и подготовленных копий и правила удаления после подачи заявки.
- Передать обезличенные примеры водительских удостоверений и разрешений на работу: эти типы заявлены, но представлены недостаточно для приемочного теста.

## 21. Требования к разработке через Codex Web
- Репозиторий не содержит реальные фотографии, паспортные номера, телефоны и адреса.
- Разработка ведется небольшими PR: каркас, база, image pipeline, адаптер TSP, адаптер visitors, адаптер MGS, OCR/MRZ, UI проверки, экспорт, безопасность.
- Для тестов используются синтетические и обезличенные фикстуры.
- Windows-сборка и проверка .xls выполняются локально на Windows 11; Codex Web не является заменой тестированию на целевой ОС.
- Каждое изменение адаптера сопровождается golden-file тестом и проверкой открытия Excel без восстановления.
